# Database Standards

Standalone database conventions extracted from this codebase, reusable in any project with a relational database — regardless of the application stack on top of it. Pair this with `project-structure.md` for full-stack projects, or use it alone for backend/data services.

**Rules are written by category, not by tool.** Every rule names the *kind* of tool it governs (database, ORM, policy layer); concrete tool names appear only as examples. The reference toolchain the standards were extracted from:

| Category | Reference example |
|---|---|
| Relational database | PostgreSQL |
| ORM / schema toolkit | Prisma |
| Row-level policy layer (schema-first, on top of the ORM) | ZenStack |

**Code snippets use the reference toolchain** (the schema examples are ZenStack's `zmodel` DSL). Read them as illustrations of the rule; translate the idiom to your project's equivalent — the principle transfers even when the tool does not.

---

## 1. Schema Organization

- **A single annotated schema file is the source of truth** for models *and* access policies (e.g. `schema.zmodel`; a plain ORM's schema file plays the same role minus policies). All ORM artifacts (client, migration schema, typed query hooks) are generated from it.
- **Never edit generated schema files.** Change the source model and regenerate; wire the generate step into the build (`build = generate + compile`).
- **Abstract base models capture shared columns** — every model extends a `Base` with a string cuid `id`, `createdAt`, and `updatedAt` (use your schema DSL's inheritance or composition mechanism):

```zmodel
abstract model Base {
  id        String   @id @default(cuid())
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

- Shared field groups beyond the base (e.g. a set of analysis parameters used by several models) also become abstract models rather than being copy-pasted.

## 2. Naming

| Element | Convention | Example |
|---|---|---|
| Models | `PascalCase`, singular | `TradeSession` |
| Fields | `camelCase` | `quarterlyAmount` |
| Foreign keys | `<model>ID` (uppercase `ID` suffix) | `userID` |
| Relation lists | plural camelCase | `paperTrades`, `subscriptions` |
| Enums | `PascalCase` name, `UPPER_SNAKE` values | `PaperTradeStatus.ACTIVE` |
| Env vars | `UPPER_SNAKE_CASE` | `DATABASE_URL`, `DIRECT_URL` |

## 3. Column Type Rules

- **Money and precision-sensitive quantities use the exact-decimal column type** (`Decimal`/`numeric`), never a float type. Application code mirrors this with a decimal library — raw floating-point arithmetic on these values is banned.
- **Timestamps are datetime columns normalized to UTC** before writing; formatting to local time happens only at the display layer.
- **Nullable means "legitimately absent"**, not "not yet migrated" — fields that always exist for new rows get defaults instead of `?`.
- **IDs are collision-resistant string IDs (e.g. cuid)**, generated by the schema default, never by application code.

## 4. Authorization

- **Authorization lives in the schema as row-level policies** (via the policy layer, e.g. ZenStack), not scattered through query code:

```zmodel
model PaperTrade extends Base {
  userID String
  ...
  @@allow('create', auth() != null)
  @@allow('read,update,delete', auth().id == userID || auth().role == "admin")
}
```

- If the ORM has no policy layer, centralize the equivalent ownership checks in **one** place (a scoped client factory or repository), never per-query.
- **Two client tiers, named to enforce intent:**
  - A **policy-enforced client factory bound to the current user** — the default for **all** application code (reference naming: `getPrisma(user)`; name yours after your ORM). It is injected into request handlers via the request context (e.g. `locals.db`) so handlers never construct their own.
  - The raw, policy-free client is exported under a deliberately hostile name with a deprecation marker, so misuse is visible in review and as editor strikethroughs:

```ts
/**
 * @deprecated dangerousPrismaDoNotUse does not have any authorization checks.
 * Always use getPrisma unless you are 100% sure you know what you are doing.
 */
export const dangerousPrismaDoNotUse = prismaClient;
```

- Legitimate raw-client use is limited to system-level code: seeds, migrations, background jobs acting as the system, and auth bootstrap.

## 5. Query Patterns

- **Reusable query shapes live in a `selectors.ts` module**, typed with `satisfies` against the ORM's arg types so the literal shape is preserved while being checked (reference ORM shown):

```ts
export const userSelector = {
  include: {
    userConfig: true,
    subscriptions: { where: {...}, orderBy: { createdAt: 'desc' as const }, take: 1 }
  }
} satisfies Prisma.UserDefaultArgs;
```

- **No repository pattern** — the service layer calls the ORM directly. The policy layer makes a defensive repository wrapper redundant.
- **Ownership filters are still written explicitly** in queries (`where: { userID: user.id }`) even though policies would enforce them — the policy is the safety net, not the query.
- Client construction happens **once**, at module level, with log verbosity switched by a debug env flag; connect at startup, not per-request.
- **ORM-level middleware/extensions absorb cross-cutting data quirks** (e.g. cleaning up conflicting rows before insert) so the workaround lives in one place with a comment explaining **why**.

## 6. Transactions

- Explicit transactions are used **sparingly**, only where multi-step consistency genuinely demands them (e.g. auth flows that create several rows atomically).
- Batch maintenance work prefers per-item error isolation (try/catch per item, continue on failure, report totals) over one giant transaction.

## 7. Migrations & Environments

- Schema changes flow: edit source schema → generate → migrate, wrapped in package scripts (e.g. `db:generate`, `db:migrate`, `db:push`). Migrations are committed.
- Two connection URLs where the platform requires it (e.g. serverless/pooled Postgres): pooled (`DATABASE_URL`) for the app, direct (`DIRECT_URL`) for migrations.
- Connection secrets live in git-ignored per-environment `.env` files; `.env.example` documents every required variable.
- Seed scripts are standalone entry points (`scripts/seed/`), run explicitly, and are the one sanctioned home of the raw client.

## 8. Anti-Patterns

- ❌ **Using the raw, policy-free client in application code.**
- ❌ **Importing the ORM package in UI code** (lint-enforced). UI imports re-exported types from the central types module.
- ❌ **Editing generated schema/client files.**
- ❌ **Float columns (or raw `number` math) for money.**
- ❌ **Duplicating an include/select shape inline** that already exists as a shared selector.
- ❌ **Per-query hand-rolled authorization** when a policy (or centralized check) should own it.
- ❌ **Storing local-time datetimes** or formatting dates before persistence.
